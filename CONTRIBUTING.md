# Contributing to Recohut Labs

Thank you for helping make experimental software easier to understand and reproduce.

## Before opening a change

1. Read the target repository's README and local contribution guidance.
2. Search existing issues and pull requests before starting overlapping work.
3. Open an issue for substantial changes so scope and evidence can be agreed first.

## Pull-request expectations

A useful pull request:

- solves one clearly described problem;
- includes or updates tests and reproducible evidence;
- documents changed behavior and known limitations;
- avoids unrelated cleanup;
- contains no secrets, private data, or third-party material without permission.

Experimental projects still need honest claims. If a workflow is deterministic, describe it as deterministic. Reserve stronger capability labels for implementations that contain the required runtime behavior and evidence.

By participating, you agree to follow the [Code of Conduct](CODE_OF_CONDUCT.md).

# Recohut Labs brand direction

## Problem statement

How might we establish Recohut Labs as the trusted open-source experimentation arm of Recohut, where engineers can discover, run, and contribute to practical AI and data systems?

## Recommended direction

Recohut Labs inherits the established Recohut mark and adds a distinct editorial identity: dark technical surfaces, the existing Recohut green, precise language, and visible project-maturity boundaries. Like AWS Labs, the organization should gain credibility from a clear parent-brand relationship and consistently useful repositories—not from copying another company's visual system.

The brand promise is **open experiments, runnable ideas**. The audience is engineers working in applied AI and data systems. The first measure of success is credibility: a visitor should understand the organization, find a project worth running, and know exactly how mature it is.

## Core identity

| Element | Standard |
| --- | --- |
| Name | Recohut Labs |
| Descriptor | The open-source experimentation arm of Recohut |
| Tagline | Open experiments. Runnable ideas. |
| Primary audience | AI and data engineers |
| Voice | Direct, technical, evidence-led, optimistic without hype |

## Visual system

| Token | Value | Use |
| --- | --- | --- |
| Recohut green | `#8DB63F` | Mark, links, active accents |
| Lab black | `#0B1220` | Primary brand surface |
| Paper | `#F7FAFC` | Text on dark surfaces |
| Steel | `#9AA8BC` | Supporting text and metadata |

Use the Recohut mark with generous clear space. Do not recolor it per project, add effects, or create project-specific versions of the parent mark. Repository artwork may introduce its own visual motif while retaining the organization lockup.

## Project maturity

Every public project must select one maturity label and define it near the top of its README:

- **Experiment** — tests a bounded idea; APIs and behavior may change without notice.
- **Reference** — documented and reproducible; intended for learning and adaptation.
- **Incubating** — maintained toward broader use; stability commitments remain repository-specific.
- **Archived** — retained for historical value and no longer actively developed.

## Minimum repository standard

Before a project is made public, it needs:

- a specific name and one-sentence GitHub description;
- a README with problem, maturity, quick start, evidence, limitations, and license;
- reproducible setup with pinned or bounded dependencies;
- an explicit security and support path;
- accurate GitHub topics and a social preview image;
- no credentials, proprietary data, or unsupported production claims.

## Key assumptions to validate

- Engineers value runnable, narrowly scoped projects more than a large mixed-content monorepo.
- A visible maturity system increases trust and sets appropriate adoption expectations.
- The Recohut parent identity already provides useful recognition and should remain prominent.

Validate these assumptions through the first three flagship releases: measure setup completion, substantive issues or pull requests, and repeat visitors—not stars alone.

## First release scope

- Complete the GitHub organization identity and public profile.
- Publish shared community health files and repository standards.
- Select three flagship projects using the release checklist.
- Give each flagship a focused repository, description, topics, maturity label, and runnable proof.

## Not doing yet

- **Bulk-moving the existing monorepo** — volume without curation would weaken the new identity.
- **Claiming production readiness** — projects earn that claim through repository-specific evidence.
- **Creating a second unrelated logo** — Labs should compound the Recohut identity.
- **Copying AWS colors or visual motifs** — the useful model is organizational clarity, not imitation.
- **Launching dozens of empty repositories** — three credible flagships are a stronger start.
